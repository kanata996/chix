# chix 项目验收标准与测试规范

本文档定义 `chix` 仓库的项目级测试质量标准。后续评审测试质量时，以本文档为主，而不是只看单个包的覆盖率，或只看单个测试文件头部清单。

适用范围：

- 根包 `chix`
- `reqx`
- `resp`
- `middleware`
- 未来新增的公开包与公开 HTTP 边界能力

## 1. 总原则

### 1.1 测试目标

单元测试的目标不是追求语句覆盖率数字，而是验证公开行为、边界条件和回归稳定性。

项目内合格测试必须满足：

- 能直接证明某个公开行为或稳定内部契约成立
- 能在缺陷存在时失败，在缺陷修复后通过
- 不依赖网络、真实时钟、随机顺序或外部环境偶然性
- 失败信息可定位具体行为，而不是只报 `got nil` / `want non-nil`

### 1.2 禁止“假测试”

以下情况不计入验收：

- 只调用函数但不验证可观察结果
- 只验证覆盖率，不验证行为
- 断言过于宽泛，多个错误实现都能误通过
- 通过读取实现细节硬凑断言，但没有验证对外契约
- 用 sleep、时序巧合、日志顺序巧合来判断结果
- 复用了 helper 覆盖，但公开包装函数自身没有直接测试

### 1.3 清单标记约定

若测试文件头部维护“测试清单”，标记语义固定如下：

- `[✓]`：该项已由真实、稳定、可复现的测试覆盖
- `[x]`：该项尚未完成，不得作为验收通过依据

文件头清单只是索引，不替代代码评审本身。评审必须继续核对对应测试实现是否真的覆盖了该项。

## 2. 通过标准

一次变更要被视为“测试质量达标”，必须同时满足以下条件。

### 2.1 基线要求

- 变更涉及的行为有直接测试覆盖
- 成功路径、失败路径、边界条件至少各有一组断言
- 回归 bug 必须先有复现测试，再有修复
- 所有新增测试本地稳定通过

### 2.2 仓库级执行要求

至少应运行：

```bash
make ci
go test ./... -race
```

若变更触及核心 HTTP 边界包或错误收敛逻辑，还应至少运行受影响包的高重复回归：

```bash
go test ./<pkg> -count=100
```

若变更触及解析、编码、归一化、错误收敛、header/query/path/body 绑定等输入面，还应评估是否需要 fuzz 测试。需要时至少补充或更新对应 fuzz case，并本地运行：

```bash
go test ./<pkg> -run=^$$ -fuzz=Fuzz -fuzztime=10s
```

说明：

- `-count=100` 不要求每个 PR 对全仓执行；至少对受影响核心包执行
- fuzz 不是每次都必须新增，但对解析器、编码器、归一化器的高风险变更，评审应明确判断“需要”或“不需要”

### 2.3 不通过条件

出现以下任一情况，应视为测试不达标：

- 公开行为变更没有测试
- 只有 happy path，没有失败和边界用例
- 只测内部 helper，未测公开入口
- 清单写成 `[✓]`，但测试实现并未覆盖对应行为
- 测试依赖随机、时序、外部服务或全局状态残留
- 断言无法防止已知缺陷回归

## 3. 各包最小验收范围

下面定义项目内各核心包在评审时必须覆盖的最小能力面。

### 3.1 根包 `chix`

根包是 facade，评审重点不是复杂逻辑，而是“导出的入口与底层包契约一致”。

至少覆盖：

- facade 是否正确暴露 `reqx` / `resp` 的核心入口
- 根包 helper 的行为是否与底层包一致
- README 中承诺的示例路径是否能被测试支撑

若 facade 只是透传，也应至少有代表性测试，防止签名、返回值或行为漂移。

### 3.2 `reqx`

`reqx` 是请求边界核心包，测试必须覆盖输入来源、顺序、校验和错误语义。

至少覆盖：

- path/query/header/body 的独立绑定
- `Bind(...)` 的绑定顺序与覆盖规则
- 空 body、错误 `Content-Type`、超大 body、未知字段
- DTO 已有值时的保留语义
- 绑定失败时不发生部分落值
- `Normalize()` 在校验前执行
- 校验错误字段名映射到请求 tag，而不是 Go 字段名
- 常见 violation 的状态码、错误码、detail 与结构化错误项
- path/query/header/body 的单源 API 与综合 API 一致性

若改动影响绑定顺序、默认值、字段覆盖或错误映射，必须补回归测试。

### 3.3 `resp`

`resp` 是响应边界核心包，测试必须覆盖成功响应、错误响应和错误收敛。

至少覆盖：

- `OK` / `Created` / `NoContent`
- `JSON` / `JSONPretty` / `JSONBlob`
- `HTTPError` 的状态码、错误码、标题、详情、公开错误项
- `WriteError(...)` 的 error 收敛和 problem JSON 写回
- 非法状态码、无响应体状态、`nil writer`、`nil request`
- pretty 行为和 raw bytes 透传行为
- 内部 cause 不泄漏到公开响应
- context canceled / timeout 的公共错误语义
- 错误响应降级路径和响应已开始写出路径
- 请求日志注解仅在约定场景出现，且不污染不该暴露的字段

若改动影响任何对外字段名、状态码、错误码、日志字段或响应体结构，必须有直接回归测试。

### 3.4 `middleware`

`middleware` 负责请求日志和相关边界中间件，测试必须覆盖日志字段与上下文行为。

至少覆盖：

- request id、trace id、route 等基础日志字段
- trace-aware 与非 trace-aware logger 的处理差异
- panic 恢复路径
- 请求上下文 logger 注入与 fallback
- 不应重复注入的字段不会重复出现
- 无 logger、无 request、无上下文等降级路径

若改动影响日志字段名、注入时机、trace 透传或 panic 恢复，必须补直接断言。

## 4. 测试设计规范

### 4.1 先测公开行为，再测内部辅助

优先顺序：

1. 公开入口
2. 公开入口依赖的关键内部契约
3. 难以通过公开入口稳定覆盖的内部 helper

不能只因为内部 helper 好测，就跳过公开入口。

### 4.2 每个核心能力至少包含三类用例

对于被视为“核心能力”的接口，至少要有：

- 正常输入下的成功行为
- 非法输入下的失败行为
- 边界条件或退化条件

示例：

- `resp.JSON(...)`
  - 成功写 JSON
  - 非法状态码报错
  - `nil writer` / bodyless status / `nil request`

- `reqx.BindBody(...)`
  - 正常 body 绑定成功
  - 非 JSON `Content-Type` 报错
  - 空 body 保持原值或 no-op

### 4.3 Bugfix 必须先有回归测试

修 bug 的正确顺序：

1. 增加能稳定复现 bug 的测试
2. 确认旧实现会失败
3. 修复实现
4. 确认新增测试通过

只改实现不补回归测试，不算完整修复。

### 4.4 日志测试必须验证“有”和“没有”

当日志字段属于契约的一部分时，既要验证应出现的字段，也要验证不应出现的字段。

例如：

- 5xx 请求日志要有 `error.code`
- 4xx 请求日志不应污染 `error.*`
- 公共错误响应不应泄漏内部 cause

### 4.5 全局状态必须成对恢复

涉及以下全局状态时，测试必须显式恢复：

- `slog.Default()`
- 进程环境变量
- 包级全局变量
- 全局时间、随机源、共享缓冲区

不能让单测之间互相污染。

### 4.6 测试命名必须描述行为

推荐格式：

- `Test<对象><行为>`
- `Test<对象><行为>When<条件>`

避免：

- `TestFoo1`
- `TestBugFix`
- `TestRandomCase`

## 5. 稳定性要求

### 5.1 必须可重复执行

测试应在本地和 CI 中稳定复现，不得依赖：

- 网络
- 外部数据库
- 实际系统时间漂移
- 未固定的随机输入
- map 遍历顺序
- goroutine 调度巧合

### 5.2 race 与高重复要求

以下变更必须特别关注稳定性：

- 并发访问
- 日志上下文传递
- 错误收敛与写回
- 共享缓冲区或共享对象复用

评审时应至少要求：

- `go test ./... -race`
- `go test ./<touched_pkg> -count=100`

### 5.3 fuzz 适用范围

以下场景优先考虑 fuzz：

- JSON/body 解码
- query/path/header 解析
- 错误归一化
- 字段裁剪与 normalize
- 原始字节透传与编码边界

若评审认为“不需要 fuzz”，应能说明原因，例如“本次只改注释，没有逻辑变化”。

## 6. 评审清单

后续审查测试质量时，至少逐项确认以下内容。

### 6.1 行为覆盖

- 公开行为是否有直接测试
- 改动点是否同时覆盖成功、失败、边界三类场景
- 是否存在只测 helper、不测公开入口的问题

### 6.2 断言质量

- 断言是否足够具体
- 失败信息是否可定位
- 是否同时验证了不应发生的行为
- 是否能真正防止该类 bug 回归

### 6.3 稳定性

- 是否有全局状态未恢复
- 是否依赖时序、随机、外部环境
- 是否适合加 `-race` / `-count` / fuzz

### 6.4 文档与清单

- 变更后的测试文件头清单是否与实现一致
- 未完成项是否明确标成 `[x]`
- 已完成项是否真实可追溯到具体测试
- 若行为契约变化，README / docs / CHANGELOG 是否需要同步

## 7. 本文档与包内清单的关系

项目级评审顺序固定如下：

1. 先看本文档，判断该类改动最低应该覆盖什么
2. 再看对应包的测试文件头清单
3. 最后核对具体测试实现是否真的覆盖清单项

也就是说：

- 本文档定义项目级标准
- 包内清单定义包级索引
- 具体测试代码才是最终证据

## 8. 当前项目推荐做法

对当前仓库，推荐采用以下实践：

- 每个核心测试文件头部维护行为清单
- 新增公开能力时，先补清单，再补测试
- 修 bug 时，先补回归测试，再修实现
- 合并前至少执行 `make ci`
- 触及核心边界包时，再执行 `go test ./<pkg> -race -count=100`
- 触及解析/归一化/编码逻辑时，评估并补充 fuzz

若未来 CI 加入更强的稳定性检查，应以 CI 为最低门槛，但本文档定义的行为验收标准仍然有效。
